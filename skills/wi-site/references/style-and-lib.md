# Wonder Lib — UI / Styling / Design System

Reference for all UI work in a Wonder site (project scaffolded from `wonder-image/new-site`). The `wonder-image/lib` package is the source of truth for design tokens, utility classes, and UI primitives. **This reference is about USING the lib, not modifying it.**

The same rules also apply when editing default components inside `wonder-image/app` (consumed by sites). When `wi-app` is the active skill, treat this file as the authoritative UI rulebook anyway.

## Authoritative sources

Always consult these in this order **before** writing CSS, JS, or new class names:

1. **`wonder-image/lib/MANIFEST.json`** — AI-readable registry of every public class, component, JS module, and entry point. https://github.com/wonder-image/lib/blob/main/MANIFEST.json
2. **`wonder-image/lib/AGENTS.md`** — agent brief: directory structure, naming conventions, build pipeline, repo-specific gotchas. https://github.com/wonder-image/lib/blob/main/AGENTS.md
3. The **lib source** at `node_modules/wonder-image/src/` (when the package is installed in the site) or the upstream repo https://github.com/wonder-image/lib.git.
4. The **compiled output** at `assets/lib/wonder-image/dist/` — read-only inspection of `backend/`, `frontend/`, `lib/`, `fonts/`, `images/`. Regenerated by `npm install` (via the package's `postinstall` copy of `dist/`).

## When this reference applies

Any task involving:

- buttons, text / typography, inputs, forms, modals, cards, alerts, badges, images
- interactive elements (dropdowns, tabs, accordions, etc.)
- default components shipped by `wonder-image/app`
- pages and components in projects scaffolded from `wonder-image/new-site`
- color / token / root-variable additions or edits
- custom CSS or JS for styling

## Hard Rules

Non-negotiable. Apply to every UI task.

- Do **not** modify `wonder-image/lib` unless the user explicitly requests it.
- Do **not** create a parallel CSS system in the project.
- Do **not** hardcode colors (HEX, RGB, HSL) in components when a token or class already exists.
- Do **not** rename existing lib classes.
- Do **not** remove existing tokens.
- Do **not** refactor unrelated code while doing UI work.
- Do **not** change JS / TS logic when the task is styling-only.
- Do **not** add new dependencies for styling-only work.
- Do **not** invent new design-system patterns when an existing utility or component already exists.
- Do **not** invent new `.wi-*` component names. `.wi-*` is the lib's naming convention for **proprietary components** — reuse the existing ones from `MANIFEST.json` (`.wi-modal`, `.wi-input-container`, `.wi-alert`, `.wi-dropdown-box`, `.wi-show`, ...). Defining a new `.wi-*` class is a lib-level change, not a site-level change.
- Do **not** hand-edit `assets/lib/wonder-image/dist/` — regenerated by `npm install` from `wonder-image/lib`.
- Do **not** treat `node_modules/wonder-image/dist/` as editable source.

## Naming conventions (from the lib)

| Prefix | Meaning | Examples |
| --- | --- | --- |
| (none) | utility class — kebab-case, atomic | `.f-end`, `.col-3`, `.bg-cover`, `.tx-upper`, `.mt-5` |
| `wi-` | **proprietary component** | `.wi-modal`, `.wi-input-container`, `.wi-alert`, `.wi-dropdown-box`, `.wi-show` |
| `btn-`, `badge-` | Bootstrap-compatible | `.btn-primary`, `.badge-icon` |
| `*-p-*` | phone variant (≤768px) | `.col-p-1`, `.f-p-end`, `.d-p-column` |
| `*-t-*` | tablet variant (≤992px) | `.col-t-1`, `.f-t-end` |
| `.pc-none`, `.tablet-none`, `.phone-none` | breakpoint-based hide | hide on desktop / tablet / phone |

Breakpoints: phone ≤768px, tablet ≤992px, desktop >992px. Some legacy resize utilities use 1000px for tablet.

## USE vs MODIFY

First step on any UI task is to classify it.

- **USE** — consume existing lib classes, tokens, and components in project markup. This is the default mode.
- **MODIFY** — touch lib internals, rename classes, add new lib-level tokens or primitives, define new `.wi-*` components. Allowed **only** on an explicit user request.

If a task seems to require modifying the lib, first attempt reuse or composition of existing primitives. If still impossible, stop and ask the user before changing the lib.

## Reuse-First Workflow

1. Classify the request: USE the lib or MODIFY it.
2. If USE: do **not** modify `wonder-image/lib`.
3. Inspect existing lib classes, tokens, and components. In order:
   1. Read `wonder-image/lib/MANIFEST.json` — the registry. Find an entry that matches the UI you need.
   2. Read `wonder-image/lib/AGENTS.md` for naming conventions and directory structure.
   3. Drill into the source under `node_modules/wonder-image/src/build/{frontend,backend,global}/...` if the manifest entry is not enough.
   4. Otherwise inspect the compiled output under `assets/lib/wonder-image/dist/` (read-only).
4. Pick existing classes and tokens that match the target UI. Use their existing names as-is — no renaming, no prefix wrappers.
5. Only when no existing class or token solves the problem, propose the **minimal** addition:
   - new custom CSS → `assets/{ASSETS_VERSION}/css/`
   - new custom JS → `assets/{ASSETS_VERSION}/js/`
   - new color token → `assets/{ASSETS_VERSION}/css/set-up/color.css`
   - new root-level token → `assets/{ASSETS_VERSION}/css/set-up/root.css`

   Always explain why the addition is necessary and why no existing primitive worked.
6. For any lib-internal change (new `.wi-*` component, new utility, new token planned for lib relocation), stop and request explicit user confirmation.

## ASSETS_VERSION Discovery

Color and root files in a Wonder site live under:

- `assets/{ASSETS_VERSION}/css/set-up/color.css`
- `assets/{ASSETS_VERSION}/css/set-up/root.css`

`{ASSETS_VERSION}` is project-specific. **Do not hardcode `0.0/` or any other version.** Detect it before reading or writing token files:

1. List `assets/` and identify the active version directory (semver-like name such as `0.0/`, `0.1/`, `1.0/`).
2. If multiple versions exist, prefer the one referenced from the project's entry layouts (`custom/view/layout/{frontend,backend}/...` and the `app/view/layout/...` framework defaults) or from the current `php forge ...` configuration.
3. If still ambiguous, stop and ask the user which version is active.

## Color and Token Rules

The lib documents tokens as **currently external** — they live in the site, not in the lib (a future refactor may consolidate them).

**Do not create a `DESIGN.md` at site root.** Visual tokens live in `assets/{ASSETS_VERSION}/css/set-up/color.css` + `root.css` and the rules in this file are the rulebook. The companion skill [`impeccable`](https://github.com/pbakaus/impeccable) accepts a `DESIGN.md` for generic projects but degrades gracefully without it — for Wonder sites it reads the real CSS plus `PRODUCT.md` (strategic context, see [`product-md.md`](product-md.md)). A second `DESIGN.md` would create a parallel token source that drifts from `color.css` / `root.css` over time; this is documented self-defense against duplication.

- Base color names (`primary`, `secondary`, etc.) can be **expanded only on explicit user request**.
- Before creating a new token, search `color.css` and `root.css` for an equivalent variable. Reuse it.
- Never hardcode HEX / RGB / HSL in a component if a token or class covers the case.
- If a new color variable is genuinely needed, add it in `assets/{ASSETS_VERSION}/css/set-up/color.css` (and `root.css` if it is a root-level token). **Do not declare it locally in a component file.**
- Some variables are read or written by the new-site backend through the `wonder-image/app` framework. Avoid solutions that lock a value statically in CSS and break that backend control path.

## Custom CSS / JS Placement

When custom CSS or JS is genuinely needed (after lib reuse has been ruled out):

- Custom CSS files → `assets/{ASSETS_VERSION}/css/`
- Custom JS files → `assets/{ASSETS_VERSION}/js/`

Do **not** place custom files inside `assets/{ASSETS_VERSION}/css/set-up/` — that directory holds the design-system token files (`color.css`, `root.css`).

## Working on `wonder-image/app` Default Components

When editing default components shipped by the framework (inside `wonder-image/app` itself):

- Apply the same reuse-first rule.
- Use lib classes instead of creating parallel CSS inside `class/App/...` or `app/...`.
- Preserve compatibility with the site token system. Default components must look correct in any site using its own `color.css` and `root.css`.
- Do not bake site-specific values into the default component.

## Required Final Report

After completing UI work, report explicitly:

- **Files modified** with paths.
- **Lib classes used** (list — both utilities and `.wi-*` components).
- **Tokens / variables used**.
- **Tokens modified or added** (if any).
- **Commands executed**.
- **Residual risks** (visual regression on other pages, dark-mode untested, responsive behavior not verified, etc.).

## Pre-Submit Checklist

Before declaring UI work done, every item must be true:

- [ ] Consulted `MANIFEST.json` before defining any new class name
- [ ] Used existing lib classes (utilities + `.wi-*` components) wherever possible
- [ ] No new `.wi-*` component invented at site level
- [ ] No parallel CSS system introduced
- [ ] No hardcoded colors in components
- [ ] No edits to `wonder-image/lib` (unless explicitly requested)
- [ ] Verified the active `assets/{ASSETS_VERSION}/css/set-up/` path when touching tokens
- [ ] No JS / TS logic changes for styling-only tasks
- [ ] No new dependencies added for styling-only work
- [ ] Diff is minimal — no unrelated refactor
- [ ] Reported files, classes, tokens, commands as listed in **Required Final Report**

## Escalation

- If the requested visual or interactive pattern is missing from the project but belongs to the shared design system, the work likely belongs in `wonder-image/lib` itself — stop and tell the user that lib modification is required, do not hack around it locally.
- If the new token would conflict with backend-managed values in new-site, surface the conflict to the user before introducing it.
